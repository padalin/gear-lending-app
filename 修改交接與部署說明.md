# 系統交接文件 — Cheerhave Gears 器材借還系統

最後更新：2026-06-21

本文件是這套系統的單一交接說明，涵蓋：系統概述、需移交的帳號、技術架構、原始碼位置、功能、日常維運、部署、安全性、已知限制。接手者讀這一份即可。

---

## 1. 系統概述

- **用途**：樂器／影音器材的借還與庫存管理。
- **使用方式**：
  - **借／還是公開、免登入**的自助流程（任何人開網站即可借、還）。
  - **管理後台需登入**，且帳號需在管理員白名單內。
- **使用者輪廓**：器材擁有者（看有什麼、在哪、偶爾借還）、資訊管理者（維護器材資料）、整理器材的協作者（需借還與完整技術資料）。

## 3. 技術架構

- 前端：React 19 + Vite 6 + Tailwind CSS（純前端 SPA）。
- 後端服務：Firebase Firestore（資料庫）、Firebase Auth（登入）、Firebase Hosting（網站）。
- 圖片：Cloudinary（瀏覽器直接上傳，unsigned preset）。
- 資料表（Firestore collections）：
  - `items`：器材主檔
  - `presets`：常用借用組合模板
  - `borrowRequests`：借用紀錄
  - `returnRecords`：歸還紀錄

## 5. 主要功能

**前台（公開）**
- 首頁器材一覽：分類顯示、搜尋、**「大圖／列表」檢視切換**（選擇記在各裝置 localStorage）。卡片顯示名稱、在庫狀態、位置，技術細節（序號/型號/顏色…）可展開。
- 借器材 / 還器材：免登入，依姓名＋電話比對。
- 器材詳情頁、櫃位地圖。

**後台（`/admin`，需登入且 email 在白名單）**
- 管理員後台：未歸還明細表（可排序）＋ 借還紀錄（詳細模式）。
- 器材管理 `/admin/items`：卡片式列表、可切換大圖/列表、單筆編輯/刪除。
- 批次管理 `/admin/bulk`：三分頁 —「批次新增」（卡片式 + CSV 匯入）、「批次編輯」（搜尋＋點開編輯）、「批次照片」（拖照片配對器材、一次上傳）。
- 模板管理 `/admin/templates`：維護常用借用組合。

## 6. 日常維運操作

### 加一個新管理員（重要：前後端兩處都要改）
1. `src/adminEmails.js` 加上 email（小寫）。
2. `firestore.rules` 的 `isAdmin()` 也加上同一個 email。
3. 重新部署（見 §8）並 `firebase deploy --only firestore:rules`。
> 只改前端 → 對方進得了後台但存不了資料；只改規則 → 反之。**務必兩邊一起改。**

### 新增 / 編輯器材
登入 → 後台 → 器材管理 → 單筆編輯，或進「批次管理」用批次新增/編輯/照片。

### 上架器材照片
後台 → 批次管理 →「批次照片」：拖一批照片進去（HEIC 會自動轉檔），用搜尋或勾選把照片指定到器材，一次上傳。

### 清理 Cloudinary 沒用到的照片（回收免費版空間）
在 App 刪照片只是移除引用，實體檔案靠這支腳本清。**密鑰只透過環境變數帶、不寫進檔案**（Cloudinary 後台 → Settings → API Keys 取得）：
```bash
# 先乾跑，只列出會刪哪些
CLOUDINARY_API_KEY=金鑰 CLOUDINARY_API_SECRET=密鑰 node scripts/cleanup-cloudinary.mjs
# 確認無誤後加 DELETE=yes 才真的刪
CLOUDINARY_API_KEY=金鑰 CLOUDINARY_API_SECRET=密鑰 DELETE=yes node scripts/cleanup-cloudinary.mjs
```

## 7. 開發 / 建置 / 部署指令

於專案根目錄執行。**npm 指令對應的環境**：
- `npm run dev` → 連**開發**資料庫 `gear-lending-dev2`（安全沙箱，測試用）
- `npm run dev:prod` → 連**正式**資料庫（會動到真實資料，少用）
- `npm run build` → 無環境變數，預設**正式**（部署用）

```bash
# 第一次取得專案
git clone https://github.com/padalin/gear-lending-app.git
cd gear-lending-app
npm install

# 本機開發測試（連測試資料庫，可手機看：加 -- --host）
npm run dev

# 部署網站到正式
npm run build
firebase deploy --only hosting        # 需先 firebase login，且帳號有 gear-lending 權限

# 部署安全規則到正式
firebase deploy --only firestore:rules

# 版本備份
git add -A && git commit -m "說明這次改了什麼" && git push
```
部署後開 https://gear-lending.web.app 驗證。

## 8. 安全性

- **雙層後台保護**：前端 `ProtectedRoute`（檢查登入 + email 白名單）＋ 後端 `firestore.rules`（真正的防線）。前端可被繞過，**規則才是關鍵**。
- **Firestore 規則重點**：器材/模板「寫入」限管理員；借/還紀錄允許公開「新增」（因借還免登入），但「修改/刪除」限管理員；讀取公開。
- **密鑰原則**：
  - Cloudinary **API Secret 絕不可放進前端或 commit**，只在本機透過環境變數給清理腳本使用。
  - Firebase 的 web 設定（apiKey 等）屬可公開識別碼，放在前端是正常的；真正的安全靠 Firestore 規則。

---

## 9. 已知限制與未來建議

- **防超賣為過渡方案**：目前借/還在送出前重新驗證庫存＋防止連點，能擋常見狀況，但非 100% 原子。要完全防併發需在 `items` 上維護庫存欄位並用 `runTransaction`，並改寫所有讀庫存處。
- **借用人個資公開可讀**：因借還免登入，姓名/電話目前任何人都讀得到。要保護需把庫存計算/歸還查詢搬到後端（Cloud Functions），屬較大改動。
- **Cloudinary 設定寫死**：cloud name / upload preset 以 fallback 寫死在程式（早期帳號搬遷未完成留下）。若日後換 Cloudinary 帳號，改 `.env` 的 `VITE_CLOUDINARY_*` 即可，不必動程式。
- **上傳未嚴格限制大小/型別**：可日後在上傳前加檢查。
