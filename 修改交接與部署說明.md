# 第三輪修改：後台「詳細模式」紀錄不再依日合併

## 問題
`AdminPage.jsx` 詳細模式原本用「日期＋類型＋借用人＋電話」分組，把同一人同一天的多次借/還併成一筆，導致同一把吉他「借→還→再借→再還」顯示成「借出 ×2、歸還 ×2」，看起來像庫存有 2 把。（庫存計算本身正確，只是這個清單合併顯示造成誤會。）

## 改法（你選的「每次送出算一筆」）
- 借用送出（`MultiBorrowForm.jsx`）時產生一個批次代號 `sessionId`，同一次借用的每一筆都帶同一個代號；歸還（`MultiReturnForm.jsx`）也寫入 `sessionId`。
- `AdminPage.jsx` 分組改為：有 `sessionId` 的（新紀錄）依 `sessionId` 合併成一張卡；沒有的（舊紀錄）改用該筆 doc id → 每筆各自一張。

## 效果
- **新紀錄**：同一次借多把 → 合併成一張卡顯示 ×N；同一天不同次借 → 各自一張卡、各自時間。完全正確。
- **舊紀錄**：因為沒有 `sessionId`，會改成「每筆原始紀錄各自一張」。所以你現在看到的「借 2 還 2」會拆成 2 筆借用、2 筆歸還、各自顯示時間，不再合併成 ×2。這是你接受的取捨（舊紀錄無法回溯補代號）。
- 庫存與總覽模式的計算不受影響，照舊正確。

## 動到的檔案
`src/MultiBorrowForm.jsx`、`src/MultiReturnForm.jsx`、`src/AdminPage.jsx`。

## 測試重點（`npm run dev`，dev 資料庫）
1. 借一款多把（需庫存足夠）一次送出 → 詳細模式應為一張卡 ×N。
2. 同一人同一天分兩次各借 1 把 → 應為兩張卡，不再合併。
3. 歸還後 → 借/還各自獨立、時間正確。

部署步驟同前（`npm run build` → `firebase deploy --only hosting`）。

# 第六輪：Firestore 安全規則（最重要的安全防線）

## 為什麼需要
前端的 admin 白名單只是「介面層」的保護。任何人只要拿到 `src/firebase.js` 裡那組設定（打包進前端，本來就看得到），就能**繞過你的網站、直接呼叫 Firestore API 讀寫資料庫**。唯一能真正擋住的是 Firestore Security Rules。我已草擬一份放在 repo 根目錄 **`firestore.rules`**。

## 規則重點
- `items`、`presets`（器材、模板）：所有人可讀；**只有管理員 email 可寫**。
- `borrowRequests`、`returnRecords`（借/還）：因為借還是公開免登入，允許任何人**讀取與新增**；但**修改/刪除只限管理員**。
- 管理員名單寫在規則裡的 `isAdmin()`，**請與 `src/adminEmails.js` 保持一致**。

## 怎麼發佈（請照做，別直接上正式）
1. **先搞清楚專案**：資料庫在**正式專案 `gear-lending`**（不是 hosting 的 `gear-lending-app`）。改規則要去 `gear-lending` 這個專案。
2. 開 Firebase Console → 選 `gear-lending-dev2`（開發）→ Firestore Database → 規則 → 貼上 `firestore.rules` 內容。
3. 用上方「Rules Playground」或直接在 dev 站點實際借一筆、還一筆、後台新增一筆，確認都正常。
4. 沒問題後，再到 `gear-lending`（正式）→ 規則 → 貼上 → **發佈**。
5. 發佈後立刻驗證：未登入借/還仍正常；未登入用工具直接寫 items 應被拒。

## 已知限制（未來再處理）
