# 器材借還 App 程式碼分析報告

分析日期：2026-06-19
範圍：`src/` 全部 31 個 JSX + 6 個 JS 檔（不含網站轉移／migration）
技術棧：React 19 + Vite + Firebase Firestore + Cloudinary

---

## 一、明顯不合邏輯（Bug，建議優先修）

### A1. `MultiItemForm.jsx` 直接報錯、無法執行【高】
`MultiItemForm.jsx:25, 31` 使用變數 `items`，但全檔從未定義、import 或從 props 接收。元件 props 只解構了 `title, submitLabel, firebaseCollection, buttonColor`。render 時會丟 `ReferenceError: items is not defined`，元件完全無法掛載。研判為已被 `MultiBorrowForm`／`MultiReturnForm` 取代的舊版半成品。建議刪除避免誤用。

### A2. 借／還沒有交易（transaction），會超賣／超還【高】
- 借用：`MultiBorrowForm.jsx:163-184` 的在庫量是「讀取當下」算出的快照，送出時只是無條件 `addDoc`，中間沒有重新驗證在庫，也沒有 `runTransaction`。兩人同時借僅剩 1 的器材時，雙方前端都顯示「在庫 1」，會各自寫入成功 → 超賣。
- 歸還：`MultiReturnForm.jsx:147-174` 同樣依舊快照驗證，可重複歸還造成超還。在庫計算 `borrowed = borrowMap − returnMap` 只有 `Math.max(...,0)` 防下限，沒有 `Math.min(total)` 防上限，超還會讓在庫被灌大於總數。
- 建議：借還改用 `runTransaction`，在交易內重新讀取並核對可借／未還數後才寫入。

### A3. 借／還身分欄位不一致，可能查無借用紀錄【高】
歸還用 `where("borrower", ...)`、`where("phone", ...)` 比對（`MultiReturnForm.jsx:53-60`）。`MultiBorrowForm` 寫入 `borrower`／`phone`（一致），但 `MultiItemForm.jsx:88-91` 寫的是 `name`／`contact`。任何經 `MultiItemForm` 寫入的借用，歸還端完全查不到。即使該檔已棄用，也凸顯欄位命名缺乏統一規範。

### A4. 寫入未 trim，精準比對易失敗【中】
`MultiBorrowForm.jsx:171-172` 寫入未 trim 的 `name`／`phone`，但歸還用原值 `where(...)` 精準比對。使用者借時輸入 `" 小明 "`、還時輸入 `小明` 就比對不到。建議寫入與查詢前統一 `.trim()`。歸還端 `MultiReturnForm.jsx:137-140` 也缺少借用端有的手機格式 `^09\d{8}$` 驗證。

### A5. `ItemCardSwipe` 顯示不存在的 `item.stock`【高】
`ItemCardSwipe.jsx:78` 直接讀 `item.stock`，但資料模型沒有此欄位（`ItemCard.jsx:55-57` 是用 `quantity − (borrowCount − returnCount)` 動態算）。若 item 無此欄位會渲染成「在庫：/0」。兩個元件對「在庫」邏輯不一致。

### A6. 在庫統計邏輯三套並存、互相矛盾【中】
- `AdminItemList.jsx:67-80` 每筆借／還紀錄一律當數量 1（`+1`/`-1`），忽略 `items[].quantity` 與陣列格式。
- `AdminPage.jsx:94-183` 則正確處理陣列與 `quantity`。
- 同一批資料兩頁算出不同庫存數字。建議抽成共用 `computeStock()` 統一處理單筆／陣列／quantity。

### A7. `BulkEditPage` 差異比對依賴 index 對齊【中】
`BulkEditPage.jsx:143-145` 用 `items.filter((item,i)=>...originalItems[i])` 依 index 一一對應；`handleChange`（`118-125`）又以 `filteredItems[index].id` 反查 index。目前順序固定尚可運作，但一旦排序／刪除或 id 重複立即失效。建議改用以 `id` 為鍵的 map 比對。

### A8. `ItemCard` 庫存色燈在數量 0 時誤判【低】
`ItemCard.jsx:56-58`：`quantity===0` 時 `inStock` 也是 0，`isFull` 為 true 顯示綠燈，把「無庫存」誤判為「滿庫」；歸還數大於借出時 `inStock` 反而超過 `quantity` 被判紅色。

### A9. 庫存／可借狀態來源不一致【中】
`ItemDetail.jsx:50-52` 用布林 `item.available`，但卡片用即時借還統計。可能出現詳情頁「可借出」、卡片「在庫 0」的矛盾。

## 二、尚未完成開發

### B1. 多個「孤兒元件」未接線【中】
`Input.jsx`、`Select.jsx`、`FormCategorySection.jsx`、`MultiItemForm.jsx` 在核心流程中都沒被 import 使用。三支表單反而各自手寫重複的輸入欄位／分類折疊區塊。`components/ErrorNotifier.jsx` 定義了完整錯誤類型對照，卻沒有任何地方使用，全專案錯誤處理仍用 `alert()` 或 `setError(字串)`。建議：採用或刪除，二擇一。

### B2. 重複且命名交錯的 Cloudinary 上傳檔【低】
`uploadImageToCloudinary.js`、`uploadImageToCloudinaryWithProgress.js`、`utils/cloudinary.js` 三檔做幾乎相同的事，且函式名與檔名交叉錯置（`uploadImageToCloudinary.js` 匯出 `uploadImageToCloudinaryWithProgress`，反之亦然）。實際只用到其中一份，其餘為死碼。建議保留一份統一實作。

### B3. `ItemDetail` 頁面資訊不完整【低】
`ItemDetail.jsx:39-53` 只顯示單張圖、label、分類、描述、available，缺少卡片上有的位置／品牌／型號／序號／在庫等欄位，疑為未補齊的頁面。

### B4. 殘留開發用註解與 placeholder 文案【低】
- `main.jsx:20, 51` 殘留 `// ← 加這行`、`{/* ← 新增模板編輯頁路由 */}` 開發提示註解。
- `Success.jsx:8` 文案「有有老闆已知悉！」、`ErrorNotifier.jsx` 文案「單兵告報告」（錯字，應為「單兵報告」）疑為佔位／內部用語，正式環境建議調整。

### B5. 版本號未自動更新【中】
`version.js` 停在 `v2025-05-08_1814`，`generateVersion.js` 雖能產生 `vYYYY-MM-DD_HHMM`，但顯然未接進 build 流程，Footer 對使用者顯示的是過時版本。建議把 `generate-version` 接進 `package.json` 的 `prebuild`／CI。

## 三、需要優化

### C1. 每張卡片重複全表掃描，Firestore 讀取 N 倍放大【高，效能優先】
`ItemCard.jsx:11-53` 每張卡掛載時各自 `getDocs(borrowRequests)` + `getDocs(returnRecords)`，把整個集合抓回 client 端過濾自己那筆。一個分類 N 張卡 = N×2 次全集合下載。`App.jsx:23-67` 首頁也是抓三張完整 collection 在 client 端彙總。資料量成長後讀取成本與計費線性爆增。建議：把統計上提至父層只查一次傳入 props，或在 item 文件維護 `stock` 欄位，借還用 transaction 增減。

### C2. 圖片未用 Cloudinary 縮放、未 lazy load【中】
`ItemCard.jsx:106-111`、`ItemCardSwipe.jsx:54-61`、`ItemDetail.jsx:40-46`、`MultiBorrowForm.jsx:216,307` 等直接載入原圖，未帶 `w_400,f_auto,q_auto` 轉換參數，也沒有 `loading="lazy"`，列表頁一次下載多張大圖，行動裝置流量與渲染成本高。`ItemDetail.jsx:42` 還依賴外部 `via.placeholder.com`，該服務中斷時會破圖。

### C3. 大量整頁刷新導航，破壞 SPA 體驗【低】
`AdminItemList.jsx:112,153,170,181,189,224`、`BulkAddPage.jsx:95`、`BulkEditPage.jsx:195`、`TemplateEditorPage.jsx:128` 用 `window.location.href` / `reload()` 做整頁重載，造成白屏、重新下載 bundle、Firestore 重讀。建議改用 `useNavigate()` / `<Link>`。

### C4. 批次寫入未用 `writeBatch`，部分失敗資料不一致【高】
`BulkAddPage.jsx:75-82`、`BulkEditPage.jsx:152-155` 用迴圈逐筆 `await`，中途失敗會留下半套資料，UI 只顯示「失敗」無法得知哪些成功，重試會重複新增。借用 `MultiBorrowForm.jsx:165-177` 也是逐件 N 次 `addDoc` 序列化，非原子。建議改 `writeBatch`（單批上限 500）。

### C5. 大量重複程式碼【中】
- `ItemCard` 與 `ItemCardSwipe` 高度重複（圖片輪播 + Lightbox + 庫存）。
- `smartSort`／`catPriority` 在 `AdminPage`、`BulkEditPage`、`AdminItemList` 重複三套，且分類鍵大小寫不一致（AdminPage 用小寫 `ag/eg`，BulkEditPage 用大寫 `AG/EG`），排序結果不同。
- 三支表單重複手寫輸入欄位與在庫統計。建議抽共用元件／util。

### C6. 缺少送出中狀態，可重複點擊重複寫入【中】
`MultiBorrowForm`、`MultiReturnForm` 的送出按鈕進行中未 disabled、無 `isSubmitting`，慢網路連點會送出多份紀錄。

### C7. 非同步流程缺錯誤處理與 loading【中】
`App.jsx:23-67`（`fetchWithStock`）、`ItemDetail.jsx:16-34`、`EditItemPage.jsx:14-26` 等的 Firestore 讀取無 try/catch、無 loading／notFound 狀態。失敗或慢網路時畫面白屏或永久卡在「載入中或找不到」，使用者無回饋。

### C8. 動態 Tailwind class 會被 purge【低】
`MultiItemForm.jsx:193` 用 `bg-${buttonColor}-600` 拼接 class，Tailwind 編譯期掃不到 → 樣式失效。建議改 colorMap 對照表或 safelist。

### C9. 無障礙（a11y）普遍缺失【中】
- 彈窗（`ItemCard`、`ItemCardSwipe`、`CabinetMap`、`Modal`）皆無 Esc 關閉、focus trap、`role="dialog"`；`Modal.jsx` 連點背景關閉都沒有。
- icon／符號按鈕（地圖切換、`＋/−` 數量、折疊 `▼/▶`、刪除「x」）缺 `aria-label`、`aria-expanded`。
- `Login.jsx:34-49` 欄位只有 placeholder 無 `<label>`，password 缺 `autoComplete`。
- `ImageUploader.jsx:120-132` 上傳區是 `div` 加 onClick，無 role／鍵盤支援。

### C10. 列表用 array index 當 key【低】
多處 `key={idx}`（`ItemCard`、`ItemCardSwipe`、`AdminPage:447,464,474`、`ImageUploader:148`、`ImageEditorCell:74`、`TemplateEditorPage:246`、`Select:16`）。可拖曳重排的圖片清單尤其會造成 DOM 重用錯誤、預覽閃爍。建議用 url／id 當 key。

### C11. 其他【低】
- `App.jsx:77` `filterItem` 多數欄位未 `|| ""`，undefined 會插入字面 `"undefined"` 污染搜尋。
- `NewItemForm.jsx:32`、`ItemForm.jsx:30` quantity 用 `parseInt(value)` 未防 NaN（清空時變 NaN）。`TemplateEditorPage.jsx:181-186` 數量無下界保護。
- `MultiBorrowForm.jsx:134-143` 套用模板數量未受在庫上限限制（缺 `Math.min`）。
- `MultiReturnForm.jsx:40-92` 查詢 `useEffect` 無 debounce、無取消舊請求，快速輸入時舊回應可能覆寫新結果。
- timestamp 用法不一（`new Date()` vs `Timestamp.now()`），建議統一 `serverTimestamp()`。
- `ImageEditorCell.jsx:14-34` 無 HEIC 轉檔、無大小門檻、單一 try/catch 任一檔失敗整批中斷，與 `ImageUploader` 不一致。

## 四、其他提醒（安全／架構，最高優先）

### S1. 後台路由幾乎沒有保護【高，最優先】
`main.jsx` 中只有 `/admin`（`40-47`）被 `<ProtectedRoute>` 包住，其餘全部裸露：`/admin/bulk-edit`(31)、`/admin/bulk-add`(38)、`/admin/items`(48)、`/admin/items/new`(49)、`/admin/items/edit/:id`(50)、`/admin/templates`(51)。未登入者直接輸入網址即可進入新增、編輯、批次編輯頁。建議用巢狀路由 `<Route path="/admin" element={<ProtectedRoute><Outlet/></ProtectedRoute>}>` 統一保護。

### S2. 「登入」不等於「管理員」【高】
`ProtectedRoute.jsx:9` 只檢查 `if (!user)`，沒有任何角色判斷。只要是有效 Firebase 帳號（若開啟自助註冊更嚴重）就能進後台執行所有管理操作。建議用 Firebase Custom Claims（`admin:true`）或 Firestore 白名單。

### S3. Firebase 與 Cloudinary 設定硬寫進原始碼【高】
- `firebase.js:6-22` dev／prod 兩組 config 明文寫死並一起打包進 bundle。
- `uploadImageToCloudinary.js:5-6,29-30`、`uploadImageToCloudinaryWithProgress.js:5-6`、`utils/cloudinary.js:2-3` 把真實 `CLOUD_NAME`／`UPLOAD_PRESET` 當 fallback 寫死。任何人都能用這組 unsigned preset 往你的 Cloudinary 帳號上傳垃圾、耗用配額。
- 建議：移到 `.env`、移除硬寫 fallback；Cloudinary preset 設資料夾／大小／格式限制，考慮改 signed upload。

### S4. 真正的防線在 Firestore Security Rules【高，務必確認】
以上前端守衛只是 UX。若 Firestore Security Rules 寬鬆（例如 `allow read, write: if true`），任何人拿到 `firebase.js` 那組 config 就能直接讀寫正式資料庫，前端做再多都無效。**請務必確認 `items`／`borrowRequests`／`returnRecords`／templates 等 collection 的寫入規則限制為管理員，並啟用 App Check。**（本次審查未含 rules 檔，無法判斷，但這是最關鍵的安全防線。）

### S5. 上傳無大小／型別實質驗證【中】
`accept="image/*"` 只是檔案選擇器提示，拖曳仍可放任意檔。`ImageUploader.jsx`、`ImageEditorCell.jsx` 未設大小上限，超大檔在壓縮階段可能讓行動瀏覽器當掉。建議在上傳前過濾 MIME 與大小。

## 修復優先序建議

1. **最優先（安全）**：S1 補齊所有 `/admin/*` 路由守衛、S2 加 admin 角色判斷、S4 確認並收緊 Firestore Rules、S3 移除硬寫密鑰。
2. **次優先（資料正確性）**：A2 借還改 transaction 防超賣超還、A3 統一身分欄位、A1 刪除壞掉的 MultiItemForm、A6 統一在庫統計。
3. **效能**：C1 消除每卡全表掃描、C4 批次改 writeBatch、C2 圖片優化。
4. **品質與體驗**：孤兒元件清理、重複碼抽共用、a11y、loading/錯誤處理、SPA 導航。
